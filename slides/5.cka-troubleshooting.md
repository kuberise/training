## CKA Exam Overview

The **Certified Kubernetes Administrator (CKA)** validates hands-on cluster administration skills.

* **Format**: Performance-based, terminal environment, ~2 hours.
* **Open book**: [kubernetes.io/docs](https://kubernetes.io/docs/) is allowed.
* **Focus areas**: Cluster architecture, workloads, services, storage, troubleshooting.

### Preparation Resources
* **[Kodekloud](https://kodekloud.com/)** — structured CKA course and labs.
* **[Killer.sh](https://killer.sh/)** — official exam simulator (included with exam purchase).
* **`kubectl explain`** — your best friend during the exam.

## Exam Tips & Efficiency

* **Verify your work**: Always run `kubectl get` after creating resources.
* **Use aliases**: `alias k=kubectl` and `alias kgp="kubectl get pods"`.
* **Set namespace**: `kubectl config set-context --current --namespace=kube-system`.
* **Vim basics**: Know how to edit YAML in the exam terminal (`i`, `Esc`, `:wq`).
* **Imperative shortcuts**: `kubectl create`, `kubectl run`, `kubectl expose` save time.
* **Bookmark docs**: Pod, Deployment, Service, RBAC, NetworkPolicy reference pages.

## Command-Line Efficiency

### Essential kubectl Patterns

```bash
# Dry-run to generate YAML quickly
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml

# Watch resources in real time
kubectl get pods -w

# Sort and filter
kubectl get pods --sort-by=.metadata.creationTimestamp
kubectl get pods --field-selector=status.phase=Pending

# Debug a failing Pod
kubectl describe pod <name>
kubectl logs <pod> -c <container> --previous
kubectl exec -it <pod> -- /bin/sh
```

## Cluster Bootstrapping with kubeadm

`kubeadm` is the standard tool for creating production-ready Kubernetes clusters.

### High-Level Steps
1. Install container runtime, kubeadm, kubelet, kubectl on all nodes.
2. Initialize control plane: `kubeadm init`.
3. Install CNI network plugin.
4. Join worker nodes: `kubeadm join <control-plane>:6443 --token ...`.
5. Configure kubectl for the admin user.

```bash
kubeadm init --pod-network-cidr=10.244.0.0/16
kubeadm token create --print-join-command
```

## Cluster Upgrades

Upgrade order: **control plane first**, then worker nodes. Always upgrade one minor version at a time.

### Control Plane Upgrade
```bash
# Drain node (for workers)
kubectl drain <node> --ignore-daemonsets

# Upgrade kubeadm
apt-get update && apt-get install -y kubeadm=<version>

# Plan and apply upgrade
kubeadm upgrade plan
kubeadm upgrade apply v1.xx.x

# Upgrade kubelet and kubectl
apt-get install -y kubelet=<version> kubectl=<version>
systemctl daemon-reload && systemctl restart kubelet

# Uncordon node
kubectl uncordon <node>
```

## etcd Backup & Restore

etcd stores all cluster state. Back it up before upgrades or major changes.

### Backup (on control plane node)
```bash
ETCDCTL_API=3 etcdctl snapshot save /tmp/etcd-backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

### Restore
```bash
ETCDCTL_API=3 etcdctl snapshot restore /tmp/etcd-backup.db \
  --data-dir=/var/lib/etcd-restore
# Update etcd manifest to point to restored data-dir, restart etcd
```

## Node Troubleshooting

When a node is **NotReady**:

```bash
kubectl get nodes
kubectl describe node <node-name>
systemctl status kubelet          # On the node
journalctl -u kubelet -f          # Kubelet logs
```

### Common Causes
* Kubelet not running or misconfigured.
* Container runtime (containerd) not running.
* Disk pressure or insufficient resources.
* Network/CNI plugin failure.
* Certificate expiration.

## Pod & Workload Troubleshooting

Systematic approach for failing Pods:

```bash
kubectl get pods -o wide
kubectl describe pod <pod-name>     # Events section is key
kubectl logs <pod-name>             # Current logs
kubectl logs <pod-name> --previous    # Crashed container logs
kubectl get events --sort-by=.lastTimestamp
```

### Common Pod States
| State | Likely Cause |
|-------|-------------|
| **Pending** | Insufficient resources, scheduling constraints, PVC not bound |
| **CrashLoopBackOff** | Application error, missing config/secret, wrong command |
| **ImagePullBackOff** | Wrong image name/tag, missing registry credentials |
| **OOMKilled** | Memory limit exceeded |
| **Evicted** | Node resource pressure |

## JSONPath & Output Formatting

Extract specific fields efficiently — essential for the CKA exam.

```bash
# Get Pod IP
kubectl get pod nginx -o jsonpath='{.status.podIP}'

# Get all container images in a namespace
kubectl get pods -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[*].image}{"\n"}{end}'

# Custom columns
kubectl get pods -o custom-columns=NAME:.metadata.name,STATUS:.status.phase,NODE:.spec.nodeName

# Export resource as YAML for editing
kubectl get deployment nginx -o yaml > nginx.yaml
```

### Useful jsonpath Paths
* `.metadata.name`, `.metadata.namespace`
* `.status.phase`, `.status.podIP`
* `.spec.containers[*].image`
* `.spec.nodeName`

## Troubleshooting with k9s

Use k9s for real-time cluster observation during labs and exams.

* `:pods` — filter and inspect Pods
* `d` — describe, `l` — logs, `e` — edit YAML
* `:xray deploy` — visual dependency graph
* Shift+0 — show all namespaces
