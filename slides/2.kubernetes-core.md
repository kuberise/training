## Why Containers & Orchestration?

* **Containers** package code, runtimes, system tools, and settings together.
  * **Portable**: Run anywhere (laptop, cloud, on-premises).
  * **Consistent**: Eliminates configuration drift.
  * **Scalable**: Scale instances up or down easily.
* **The Challenge**: How do you manage 100+ containers across 10+ servers?
* **The Solution**: **Kubernetes** (container orchestration).

---

## What is Kubernetes?

Kubernetes (K8s) is an open-source platform for automating deployment, scaling, and management of containerized applications.

* Service discovery and load balancing
* Storage orchestration
* Automated rollouts and rollbacks
* Automatic bin packing (resource optimization)
* Self-healing (restarts failed containers, replaces pods)
* Secret and configuration management

**Cluster lifecycle**


## Kubernetes Architecture

```
                  ┌────────────────────────────────────┐
                  │           CONTROL PLANE            │
                  │  apiserver, etcd, scheduler, c-m   │
                  └──────────────────┬─────────────────┘
                                     │
                  ┌──────────────────┴─────────────────┐
                  │                                    │
         ┌────────┴──────────┐              ┌──────────┴────────┐
         │    WORKER NODE    │              │    WORKER NODE    │
         │  kubelet, proxy   │              │  kubelet, proxy   │
         └───────────────────┘              └───────────────────┘
```

---

## Control Plane Components

* **API Server (`kube-apiserver`)**: Front-end for the control plane; all communication goes through it.
* **etcd**: Highly-available key-value store; backing store for all cluster data.
* **Scheduler (`kube-scheduler`)**: Selects the best node for unscheduled Pods.
* **Controller Manager (`kube-controller-manager`)**: Runs controllers (Node, Job, Endpoint) to maintain desired state.

---

## Data Plane (Worker Node) Components

* **Kubelet**: Node agent; ensures containers run according to PodSpec.
* **Kube Proxy (`kube-proxy`)**: Maintains network rules; enables Pod communication.
* **Container Runtime**: Runs containers (e.g., `containerd`, CRI-O).

## kubectl: The Kubernetes CLI

`kubectl` communicates with the cluster API server.

* **Configuration**: `~/.kube/config` (override with `KUBECONFIG`).
* **Context**: Combination of Cluster, User, and Namespace.
* **Helper Tools**: `kubectx` (switch contexts), `kubens` (switch namespaces).

```bash
kubectl cluster-info
kubectl config get-contexts
kubectl get nodes
kubectl get pods --all-namespaces
kubectl api-resources -o wide
kubectl explain pod.spec.containers
```

## Kubeconfig: Precedence & Safer Practices

**Precedence** (highest wins):

1. `--kubeconfig` flag on the command
2. `KUBECONFIG` environment variable
3. `~/.kube/config` (default fallback)

**Safer practice**: do not pile every cluster (especially production) into the default `~/.kube/config`. One accidental `kubectl delete` in the wrong context can break prod. Keep each cluster in its own file and opt in explicitly:

```bash
export KUBECONFIG=~/.kube/dev.conf
kubectl get pods
```

**Merge multiple files** with `:` (Linux/macOS) or `;` (Windows). `kubectl` treats them as a single merged view:

```bash
export KUBECONFIG=~/.kube/dev.conf:~/.kube/test.conf
kubectl config get-contexts
kubectl config use-context dev
```

## Sample Kubeconfig: Clusters + Users + Contexts

A **context** is a named tuple of `(cluster, user, namespace)`. Mix and match freely.

```yaml
apiVersion: v1
kind: Config
current-context: dev-alice

clusters:
- name: dev-cluster
  cluster:
    server: https://dev.k8s.example.com
    certificate-authority-data: <base64-ca>
- name: prod-cluster
  cluster:
    server: https://prod.k8s.example.com
    certificate-authority-data: <base64-ca>

users:
- name: alice
  user:
    client-certificate-data: <base64-cert>
    client-key-data: <base64-key>
- name: bob-readonly
  user:
    token: <bearer-token>

contexts:
- name: dev-alice           # alice  + dev-cluster  + team-a
  context: {cluster: dev-cluster,  user: alice,        namespace: team-a}
- name: dev-bob             # bob    + dev-cluster  + team-b
  context: {cluster: dev-cluster,  user: bob-readonly, namespace: team-b}
- name: prod-alice          # alice  + prod-cluster + team-a (same user, different cluster)
  context: {cluster: prod-cluster, user: alice,        namespace: team-a}
```

Same user across clusters, same cluster with different users, same pair with different namespaces, all just contexts.

## Useful `kubectl config` Commands

**Inspect**

```bash
kubectl config view                              # full merged config (redacts secrets)
kubectl config view --minify                     # only the current context
kubectl config view --raw                        # include cert/key data
kubectl config get-contexts                      # list contexts; * marks active one
kubectl config current-context                   # print the active context name
```

**Switch**

```bash
kubectl config use-context dev-alice             # switch active context
kubectl config set-context --current --namespace=team-b   # change default ns
kubectl --context=prod-alice get pods            # one-shot override, no switch
```

**Create / edit / remove**

```bash
kubectl config set-context dev-bob \
  --cluster=dev-cluster --user=bob-readonly --namespace=team-b
kubectl config rename-context dev-alice alice@dev
kubectl config delete-context old-ctx
kubectl config unset users.bob-readonly          # remove a user entry
```

Tip: `kubectx` / `kubens` wrap these into one-keystroke pickers, and most shells have a prompt plugin (`kube-ps1`, starship) that shows the active context, a cheap safety net against running prod commands by accident.

## Imperative vs. Declarative

### Imperative (Direct Instructions)
* "Run this container, scale this deployment to 3."
* Example: `kubectl run nginx --image=nginx`
* Good for quick testing; hard to version control.

### Declarative (Desired State)
* "Here is a file describing what the system should look like. Make it so."
* Example: `kubectl apply -f deployment.yaml`
* Recommended for production; fits perfectly with GitOps.

---

## Imperative Commands Practice

```bash
kubectl run my-pod --image=nginx
kubectl create deployment my-deployment --image=nginx --replicas=3
kubectl scale deployment my-deployment --replicas=5
kubectl expose deployment my-deployment --type=NodePort --port=80
kubectl create secret generic my-secret --from-literal=password=secretpass
kubectl create configmap my-config --from-literal=port=8080
```

*Run any command twice — the second attempt returns "already exists". With `kubectl apply`, the second run reports "unchanged".*

## Imperative + `--dry-run=client -o yaml`: Bootstrapping Manifests

Hand-writing a Deployment from a blank file is painful. Let `kubectl` draft it for you, then edit and commit.

```bash
# generate, don't apply
kubectl run my-pod --image=nginx \
  --dry-run=client -o yaml > pod.yaml

kubectl create deployment web --image=nginx --replicas=3 \
  --dry-run=client -o yaml > deployment.yaml

kubectl create service clusterip web --tcp=80:8080 \
  --dry-run=client -o yaml > service.yaml

kubectl create configmap app-config --from-literal=PORT=8080 \
  --dry-run=client -o yaml > configmap.yaml

kubectl create secret generic app-secret --from-literal=PASSWORD=s3cret \
  --dry-run=client -o yaml > secret.yaml
```

* `--dry-run=client` builds the object locally, never hits the API server.
* `-o yaml` prints it instead of applying it.
* Tidy it up (drop `creationTimestamp: null`, `status: {}`), commit, then `kubectl apply -f`.
* Same trick exports a live object: `kubectl get deploy web -o yaml > deployment.yaml`.

## Kubernetes Resource Structure

Every YAML manifest has four mandatory top-level fields:

1. **`apiVersion`**: Kubernetes API schema version.
2. **`kind`**: Resource type (`Pod`, `Deployment`, `Service`).
3. **`metadata`**: Name, namespace, labels.
4. **`spec`**: Desired state of the resource.

---

## Core Resource: Pods

The smallest deployable unit in Kubernetes.

* Represents a single instance of a running process.
* Can contain **one or more containers** sharing network and storage.
* **initContainers**: Run before main containers start.
* **Sidecar pattern**: Supporting containers (log shippers, proxies).

[Hands-on: `1.kubernetes/1.pod.yaml`]

---

## Resource Requests & Limits

* **Requests**: Minimum CPU/Memory guaranteed; used by the scheduler.
* **Limits**: Maximum CPU/Memory allowed.
  * **CPU**: Millicores (`250m` = 0.25 core); exceeding limits → throttling.
  * **Memory**: Bytes (`64Mi`, `1Gi`); exceeding limits → **OOMKilled**.

---

## Core Resource: Deployments

Manages stateless Pods.

* Declares the desired number of replicas.
* Handles rolling updates (zero-downtime) and rollbacks.
* Automatically replaces failed Pods.

[Hands-on: `1.kubernetes/3.1.deployment.yaml`]

---

## Core Resource: Services

Provides a stable network endpoint (IP and DNS) for a set of Pods.

* **ClusterIP** (Default): Internal cluster communication.
* **NodePort**: Static port (`30000-32767`) on each Node's IP.
* **LoadBalancer**: External load balancer in cloud environments.
* **ExternalName**: Maps to an external DNS name.
* **Headless** (`clusterIP: None`): Returns direct Pod IPs via DNS.

[Hands-on: `1.kubernetes/6.service.yaml`]

---

## ConfigMaps & Secrets

Decouples configuration from container images.

* **ConfigMaps**: Non-confidential key-value configuration.
  * Injected as environment variables, args, or mounted files.
  * [Hands-on: `1.kubernetes/9.configmap.yaml`]
* **Secrets**: Sensitive data (passwords, tokens); base64 encoded, mounted in `tmpfs`.
  * [Hands-on: `1.kubernetes/12.secret.yaml`]

---

## Namespaces & Labels

* **Namespaces**: Virtual partitions within a cluster (`dev`, `test`, `prod`).
  * [Hands-on: `1.kubernetes/15.namespace.yaml`]
* **Labels**: Key-value pairs attached to resources for organization.
* **Selectors**: Queries used by Services and Deployments to target Pods.

```yaml
metadata:
  labels:
    app: nginx
    environment: production
```

---

## Scaling Terminology

* **Scale Out / Scale In** (Horizontal): Add or remove Pod replicas.
* **Scale Up / Scale Down** (Vertical): Adjust CPU/Memory for existing containers.
