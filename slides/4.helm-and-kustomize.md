## The Manifest Management Challenge

As applications grow and deploy to multiple environments (Dev, Test, Prod), managing raw YAML manifests becomes painful:

* **Duplication**: Copy-pasting manifests leads to configuration drift.
* **Hardcoded Values**: Hard to change image tags, replicas, or hostnames per environment.
* **No Versioning**: Hard to package, share, and rollback deployments.

### Two Solutions
1. **Helm**: Templating and packaging (parameterized YAML).
2. **Kustomize**: Template-free overlaying (patching raw YAML).

## Helm: The Package Manager

Helm packages Kubernetes resources into reusable **Charts**.

* **Chart**: Bundle of pre-configured Kubernetes templates.
* **Values**: Configuration parameters (`values.yaml`) that customize templates.
* **Release**: A running instance of a Chart in a cluster.
* **Repository**: Server for sharing Charts (e.g., [Artifact Hub](https://artifacthub.io)).

## Essential Helm Commands

```bash
helm repo add nginx-stable https://helm.nginx.com/stable
helm repo update
helm search repo nginx
helm show values nginx-stable/nginx-ingress

helm install my-release nginx-stable/nginx-ingress
helm upgrade my-release nginx-stable/nginx-ingress --set controller.replicaCount=2

helm history my-release
helm rollback my-release 1
helm uninstall my-release
```

## Kustomize: Template-Free Customization

Kustomize is built into `kubectl` (`kubectl apply -k`).

* **No Templates**: Uses standard, valid Kubernetes YAML files.
* **Base & Overlays**:
  * **Base**: Common, shared manifests.
  * **Overlays**: Environment-specific folders that apply patches to the Base.
* **Kustomization File**: Defines resources and customization rules.

## Advanced Kustomize Features

* **`namePrefix`**: Prepends a string (e.g., `dev-`) to all resource names; updates references automatically.
* **`configMapGenerator`**: Generates ConfigMaps with content hash ‚Äî triggers rolling updates on config change.
* **JSON Patches (RFC 6902)**: Precise `replace`, `add`, `remove` operations on base manifests.
* **Label Selector Propagation**: Adds labels to metadata and selectors of Deployments/Services.

## GitOps & Kuberise.io

GitOps uses Git as the single source of truth for the desired cluster state.

### Core Principles
1. **Declarative System**: Described in manifests, Helm, or Kustomize.
2. **Git as Source of Truth**: Version-controlled desired state.
3. **Automated Delivery**: Merged changes applied automatically.
4. **Self-Healing**: ArgoCD reconciles cluster drift.

### Kuberise.io
* **ArgoCD App of Apps** pattern for the entire platform.
* Same templates for all environments; environment-specific values folders.
* Preconfigured dashboards: ArgoCD, Grafana, Keycloak.

```bash
./scripts/install.sh minikube onprem https://github.com/kuberise/kuberise.io.git main 127.0.0.1.nip.io
```
